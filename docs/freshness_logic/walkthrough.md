# 実装完了レポート：新鮮度（話題性）の逆転ロジック導入

## 実装内容
プレイヤーが開発したメニューが「奇抜（とんちき）か」「単に映えるだけか」「普通か」によって、ショーケースでの「鮮度（売上に直結するパラメーター）」の低下スピードが変化するロジックを実装しました。

### 1. 新鮮度低下ロジックの変更 (`Showcase.tsx`)
`handleOperate` 内の売上計算ロジックにおいて、メニューごとに以下の `decayRate`（減少率）を設定しました。

- **奇抜・とんちきメニュー（カルト的人気）**
  - 対象: `flameRisk >= 30` または `visualScore >= 80`
  - 減少率: `-0.15 / 日` （長持ち）
- **お洒落で映えるメニュー（量産型映え）**
  - 対象: `visualScore >= 50` かつ `flameRisk < 30`
  - 減少率: `-0.6 / 日` （あっという間に飽きられる）
- **普通のメニュー**
  - 対象: 上記以外
  - 減少率: `-0.3 / 日` （普通）

### 2. UIの「鮮度表示」の動的化
ショーケースに並んでいる各メニューの「鮮度表示（✨最高、普通、⚠️古い など）」を、単純な「作成からの経過日数」ではなく、上記の `decayRate` を加味して計算された**実際の `freshness` の数値**に基づいて判定・表示するように改修しました。
これにより、奇抜なメニューは数日経っても「✨最高」や「普通」を維持しますが、量産型映えメニューは次の日には「⚠️古い」や「廃棄寸前」になります。

## 今後の確認ポイント
- ゲーム内で意図通りに「映えメニュー」がすぐ売れなくなり、「炎上・奇抜メニュー」が長く稼いでくれるか。
- 初速と寿命のバランスが崩れてゲームが簡単すぎ（または難しすぎ）にならないか、プレイしながらの微調整が必要になる可能性があります。
