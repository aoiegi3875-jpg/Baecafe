# 奇抜な料理と映える料理の「新鮮度」分離ロジック実装計画

現状の仕様では、ショーケースに並べたメニューの「新鮮度（売上にかかる倍率）」は、メニューの種類にかかわらず「1日経過するごとに-0.3（30%減少）」という固定の仕様になっています。

これを、メニューのステータス（映え度 `visualScore` と炎上リスク `flameRisk`）に基づいて、以下のように新鮮度（話題性）の低下スピードを変えることを提案します。

## 提案する新鮮度（減少率）の仕様

メニューを以下の3つのカテゴリに分類し、それぞれ減少率を変更します。

1. **奇抜・とんちきメニュー（カルト的人気・ミーム化）**
   - **条件:** `flameRisk >= 30` または `visualScore >= 80`
   - **特徴:** 炎上や怖いもの見たさで話題が長引き、カルト的な人気を得て長期間売れ続ける。
   - **減少率:** **-0.15 / 日**（長持ちする。4日ほどは話題性が続く）

2. **お洒落で映えるメニュー（量産型映えスイーツ）**
   - **条件:** `visualScore >= 50` かつ `flameRisk < 30`
   - **特徴:** 巷に溢れる「ありふれた映え」であるため、消費スピードが異常に早く、すぐ次のトレンドに埋もれて飽きられる。
   - **減少率:** **-0.6 / 日**（1日経つと売上が半分以下になり、次々と新メニューを出さないと稼げない自転車操業になる）

3. **普通のメニュー**
   - **条件:** 上記以外（`visualScore < 50` かつ `flameRisk < 30`）
   - **特徴:** 良くも悪くも普通で、流行に左右されないコアな常連だけが頼む。
   - **減少率:** **-0.3 / 日**（現状維持）

## Proposed Changes

### Showcase Component

#### [MODIFY] Showcase.tsx (file:///c:/Users/wakar/Desktop/Bae_Cafe/src/components/game/Showcase.tsx)
- メニューごとの分類と、それに応じた `decayRate`（減少率）の計算ロジックを追加します。
- `handleOperate` 内での売上計算時、従来の `currentAge * 0.3` ではなく `currentAge * decayRate` を使用して `freshness` を算出します。
- UI上で表示している「鮮度」のテキスト（現在は `currentAge` の日数でハードコーディングされている部分）を、計算された `freshness` の数値ベース（例: 0.8以上なら最高、0.5以上なら普通、それ以下なら古い等）で動的に表示するように変更します。これにより、長持ちするメニューは2日目でも「✨最高」や「普通」と表示されるようになります。
